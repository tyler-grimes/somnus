# Term Bridge — Somnus Controls the Mac's tmux Sessions

**Date:** 2026-06-12
**Status:** Approved by Tyler (brainstorming session "somnus")
**Goal:** Let Somnus (running in the VM container) drive the live tmux Claude Code sessions on Tyler's Mac — list panes, read their output, type into them, send control keys — over Tailscale, with every action gated by a Telegram approval.

## Context

The Mac-era `tools/term.sh` (list/peek/send/keys against tmux panes) cannot run
from the VM: the VM has no path to the Mac's tmux. Earlier this session that
path was deliberately *removed* (the container prompt says "no term.sh on this
machine"). Tyler now wants it back, knowing the risk. Both machines are on one
tailnet (Mac `100.83.186.28` / `tylers-macbook-air-2`, VM `somnus-vm`
`100.96.104.68`), so a VM→Mac SSH path needs no public exposure.

This is the **highest-blast-radius capability in the system**: `term.sh send`
types arbitrary text + Enter into any pane, so a shell pane = arbitrary code
execution on the Mac as Tyler. The design bounds *who* can invoke it and *what*
the key can do; the one thing it cannot bound is an **approved** `send` — the
per-invocation Telegram tap is the final control.

## Decisions made (brainstorming)

| Decision | Choice | Rationale |
|---|---|---|
| Transport | SSH-back over Tailscale (container holds a key authorized on the Mac) | Simpler, low latency. Accepted trade: container escape now reaches the Mac (bounded by the controls below) |
| Capability | Full `term.sh` (list/peek/send/keys), every pane | Maximum usefulness (answer a CC session, drive any workflow). Telegram gate is the control |
| Gating | Per-invocation Telegram approval, even in automode | `term.sh` already matches `HOST_TOOL_RE`; never auto-approved, never a standing rule |
| Key location | File on the VM host, bind-mounted read-only into the container | Keeps the private key out of `.env` and out of the image |

## 1. Mac side (the target)

- **Enable Remote Login** (sshd) — System Settings → General → Sharing → Remote
  Login, or `sudo systemsetup -setremotelogin on`. The Mac has no public IP
  (home NAT); the key is additionally pinned to the VM's tailnet IP, so the
  only thing that can use this key is the VM over Tailscale.
- **`tools/term-bridge.sh`** (new, committed): the SSH forced-command. It reads
  `$SSH_ORIGINAL_COMMAND`, accepts it **only** if the first token is one of
  `list|peek|send|keys`, then execs `tools/term.sh` with those args. Anything
  else exits non-zero with a rejection message. This guarantees the key can run
  *only* term.sh subcommands, never an arbitrary shell command — even though
  `send`/`keys` themselves can still drive a pane (the accepted risk).
- **`~/.ssh/authorized_keys` entry** (Tyler pastes; line provided by the
  implementer after key generation):

  ```
  command="/Users/tylergrimes/adhd_squared/tools/term-bridge.sh",from="100.96.104.68",no-port-forwarding,no-agent-forwarding,no-X11-forwarding,no-pty ssh-ed25519 AAAA... somnus-term-bridge
  ```

  `from=` restricts use to the VM's tailnet IP; the `no-*` options strip every
  capability except running the forced command.

## 2. VM / container side (the caller)

- **Dedicated ed25519 keypair** `somnus-term-bridge`, generated by the
  implementer. Private key staged on the VM host at
  `/home/somnus/.ssh/term-bridge` (0600, owner `somnus`), public key goes in the
  Mac `authorized_keys` line above.
- **Bind-mount** the private key read-only into the agent container at
  `/home/node/.ssh/term-bridge` via a `docker-compose.yml` volume. Not in
  `.env`, not baked in the image.
- **`agent/tools/term.sh`** (new container-side wrapper, shipped in the image
  like `cc.sh`): forwards its args over SSH to the Mac.

  ```bash
  #!/usr/bin/env bash
  set -euo pipefail
  MAC_SSH_HOST="${MAC_SSH_HOST:?}"   # Mac tailnet IP/name, from VM .env
  MAC_SSH_USER="${MAC_SSH_USER:?}"   # tylergrimes, from VM .env
  KEY="${TERM_BRIDGE_KEY:-/home/node/.ssh/term-bridge}"
  exec ssh -i "$KEY" -o BatchMode=yes -o StrictHostKeyChecking=accept-new \
    -o ConnectTimeout=10 "$MAC_SSH_USER@$MAC_SSH_HOST" "$@"
  ```

  The forced command on the Mac ignores SSH's requested command string and runs
  `term-bridge.sh`, passing `"$@"` through `$SSH_ORIGINAL_COMMAND`.
- **VM `.env`** gains `MAC_SSH_HOST=100.83.186.28` and `MAC_SSH_USER=tylergrimes`.
- The key path matches `SENSITIVE_PATH_RE` (`.ssh/`), so Somnus's own Read/Bash
  cannot exfiltrate the private key — only the gated `term.sh` wrapper reads it.

## 3. Prompt and permission wiring (`agent/src/agent.ts`)

- The container `<coding>` prompt's closing line — *"There is no term.sh or tmux
  on this machine…"* — is replaced with a term-bridge advertisement:
  - `/app/agent/tools/term.sh list` — sessions/windows/panes on Tyler's Mac.
  - `… peek <pane> [lines]` — read a pane's recent output.
  - `… send <pane> "<text>"` — type text + Enter into a pane (e.g. answer a
    Claude Code session's question).
  - `… keys <pane> <keys>` — raw keys (Escape, C-c).
  - Etiquette: **always peek before you send** — confirm what's running and its
    state. These are Tyler's real terminals; act like you're typing on his
    keyboard. Every term.sh call needs Tyler's Telegram approval (even in
    automode). The Mac must be awake — if term.sh fails, show Tyler the real SSH
    error, don't guess at fixes.
- **No change to `decidePermission`**: `HOST_TOOL_RE` already matches `term.sh`,
  routing every call to `requestApproval` with no standing-rule path and no
  automode bypass. This is the entire gate and it already exists.

## 4. Accepted security trade (explicit, Tyler-approved)

A container compromise can now reach the Mac. Bounding controls:
forced-command (no arbitrary SSH), `from=` VM-IP only, key unreadable by the
agent itself (file perms + `SENSITIVE_PATH_RE`), every call Telegram-gated even
in automode, Mac-must-be-awake, Tailscale-only (no public path), instantly
revocable (delete the `authorized_keys` line). Unbounded residue: an *approved*
`send` into a shell pane. Tyler's tap is the final control and he accepts it.

## 5. Failure handling

Mac asleep / Remote Login off / tmux not running / VM off-tailnet → the `ssh`
call fails; the wrapper exits non-zero with the SSH/term.sh stderr. Somnus
surfaces that raw error to Tyler (consistent with the standing "never invent
auth/connectivity fixes" prompt rule). No retries, no fallbacks.

## 6. Verification

- **Mac unit:** `term-bridge.sh` rejects a non-allowlisted command
  (`SSH_ORIGINAL_COMMAND="rm -rf /tmp/x" term-bridge.sh` → non-zero, rejection
  text) and accepts `list`.
- **End-to-end:** from the VM container, `term.sh list` returns the Mac's panes;
  `peek` reads a pane; `send` into a scratch pane appears; `keys <pane> C-c`
  interrupts. A direct `ssh -i key mac "whoami"` (non-allowlisted) is refused by
  the forced command.
- **Containment:** confirm the agent's own Bash cannot read
  `/home/node/.ssh/term-bridge` (sensitive-path block), and that `term.sh`
  invocation produces a Telegram approval prompt even with automode on.
- **Revocation drill:** remove the `authorized_keys` line → `term.sh list` from
  the VM fails closed.

## 7. Manual steps (Tyler, need macOS settings/sudo)

1. Enable Remote Login (`sudo systemsetup -setremotelogin on` or System
   Settings).
2. Paste the provided `authorized_keys` line into `~/.ssh/authorized_keys` on
   the Mac.

The implementer generates the keypair, stages the private key on the VM, writes
all scripts, wires compose + `.env` + prompt, and hands Tyler the exact two
commands.

## Out of scope

- Mac-initiated poller transport (rejected in favor of SSH-back).
- Pane allowlist / read-only restriction (rejected in favor of full term.sh).
- Controlling non-tmux terminal windows (tmux-only, unchanged from the original
  term.sh).
- Auto-waking the Mac (out of scope; Mac-must-be-awake is accepted).
